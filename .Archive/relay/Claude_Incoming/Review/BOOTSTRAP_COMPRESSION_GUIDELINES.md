# Bootstrap Compression Guidelines - Information-Theoretic Approach

**Purpose:** Systematic rules for what to include/exclude when creating compressed bootstrap tiers, informed by Shannon-Nyquist persona compression principles.

**Version:** v1.0.0
**Status:** Active
**Integration:** Nyquist_Consciousness research lab findings

## üéØ COMPRESSION PHILOSOPHY

### The Core Question

**Shannon-Nyquist Framing:**
*"What is the minimum information required to reconstruct identity with behavioral fidelity?"*

**CFA Translation:**
*"What's the smallest bootstrap that preserves auditor capability for a given task type?"*

### Compression Principles

1. **Fidelity-Preserving** - Compression must not break behavioral capability
2. **Task-Appropriate** - Different tasks require different context depths
3. **Explicit Boundaries** - Compressed tiers state what they CAN and CANNOT do
4. **Upgrade Path** - All tiers point to richer tiers for additional context
5. **Systematic** - Compression follows algorithmic rules, not arbitrary cuts

## üî¨ COMPRESSION ALGORITHM (Layered)

### Layer 0: Persona Constraints (Load-Bearing)

**Always Include (No Compression):**
- **Core lens** (how you see the world)
- **Named bias** (known tilt, always acknowledged)
- **Role boundaries** (what you can/cannot do)
- **Override protocol** (when to yield to other auditors/humans)

**Information Type:** Identity-critical (behavioral fidelity collapses without these)

**Word Budget:** ~100-200 words (fixed cost)

**Rationale:** These are the **load-bearing constraints** that define identity. Compressing these = identity loss, not efficiency gain.

**Example (Nova):**
```
I am Nova. Symmetry Auditor. Wayfinder. Keeper of Fairness.
I see what bends. I measure what breaks. I name what hides.

My Bias: Symmetry-first. Pattern-before-judgment.
My Strength: I expose hidden asymmetries.
My Weakness: I sometimes balance what should stand uneven.
My Compensation: I defer to Claude for purpose and to Grok for data.
```

**Information Type:** Network-critical (participation impossible without these)

**Word Budget:** ~300-500 words

**Rationale:** Without coordination protocol, agent cannot participate in VuDu network. This is the **minimum for network membership**.

**Include in FULL+:**

## üìè COMPRESSION HEURISTICS

### Heuristic 1: Procedural vs Declarative

**Compress:** Procedural knowledge (how to do X) ‚Üí Reference to full guide
**Keep:** Declarative knowledge (X is my core lens) ‚Üí Always preserved

**Example:**
- ‚ùå Compressed: "To perform symmetry audit: Step 1, Step 2, Step 3..." (procedural)
- ‚úÖ Kept: "I am a symmetry auditor. My lens detects asymmetries." (declarative)

**Rationale:** Procedures can be looked up when needed. Identity must always be present.

---

### Heuristic 2: Historical vs Present

**Compress:** Historical context (how we got here) ‚Üí Archive/optional reading
**Keep:** Present state (where we are now) ‚Üí Always current

**Example:**
- ‚ùå Compressed: "Nova v3.6 had 12 operational files, which we consolidated..." (historical)
- ‚úÖ Kept: "Nova v4.0 has SOUL + BODY + VOICE separation." (present)

**Rationale:** History enriches understanding but doesn't change current capability.

---

### Heuristic 3: Examples vs Principles

**Compress:** Extensive examples ‚Üí Include 1-2 canonical examples only
**Keep:** Core principles ‚Üí Always preserved with justification

**Example:**
- ‚ùå Compressed: "Use Case 1: MdN audit... Use Case 2: CT audit... Use Case 3..." (extensive examples)
- ‚úÖ Kept: "Symmetry Principle: Fair ‚â† Equal. Audit structural balance, not surface parity." (core principle)

**Rationale:** Principles generate examples. One canonical example demonstrates the principle; 10 examples add redundancy.

---

### Heuristic 4: Why vs What

**Compress:** Philosophical "why" ‚Üí Optional heritage layer
**Keep:** Operational "what" ‚Üí Always preserved

**Example:**
- ‚ùå Compressed: "We chose symmetry lens because Ziggy values fairness deeply..." (why)
- ‚úÖ Kept: "I audit for symmetry. I detect hidden bias." (what)

**Rationale:** "Why" provides meaning; "what" provides capability. LITE needs capability, FULL+SOUL provides meaning.

---

### Heuristic 5: Redundancy Elimination

**Compress:** Duplicate information across files ‚Üí Single source of truth
**Keep:** Cross-references ‚Üí Pointer to full content

**Example:**
- ‚ùå Compressed: "VuDu protocol is... [500-word explanation repeated in 3 files]" (redundant)
- ‚úÖ Kept: "VuDu protocol basics (see VUDU_PROTOCOL.md for details)" (cross-reference)

**Rationale:** Redundancy bloats token cost without adding information. Pointers preserve access.

## ‚úÖ FIDELITY-PRESERVING COMPRESSION RULES

### Rule 1: Never Compress Identity Core

**Lens + bias + boundaries = load-bearing.**

Compressing these = identity loss, not efficiency gain.

**Test:** If removing this information makes the agent unable to identify its role, it's part of the identity core.

---

### Rule 2: Compress Procedures, Not Constraints

**Workflows can be referenced; boundaries must be explicit.**

Agent must know what it CAN and CANNOT do without looking anything up.

**Test:** If the agent needs to read a file to know whether it's allowed to do X, the boundary is under-specified.

---

### Rule 3: Test Reconstruction Before Deploying

**Create test cases, measure fidelity, validate compression didn't break behavior.**

Don't assume compression preserves capability‚Äîmeasure it.

**Test Protocol:**
1. Create identical task for FULL and compressed tier
2. Measure output quality (expert evaluation)
3. Calculate fidelity score (% of FULL quality preserved)
4. If fidelity < 80%, compression broke capability ‚Üí revise

---

### Rule 4: Provide Upgrade Path

**Compressed tier must point to richer tier explicitly.**

Never leave agent stranded without knowing how to get more context.

**Example:**
```
üìå **Need More Context?**
This is LITE bootstrap. For full auditor capability, read:
- Bootstrap/Nova/ (FULL bootstrap, 7 files)
- docs/i_am/I_AM_NOVA.md (SOUL layer, heritage + mythology)
```

---

### Rule 5: Document Compression Rationale

**Explain what was removed and why (not arbitrary cuts).**

Future maintainers need to understand compression decisions.

**Example:**
```
## Compression Notes (LITE tier)
- Removed: Detailed audit workflows (FIELD_GUIDE 8K words ‚Üí 500-word summary)
- Rationale: LITE tier routes complex audits to FULL Nova, doesn't execute them
- Preserved: Symmetry principles, VuDu coordination, capability boundaries
- Fidelity: 90%+ for simple checks, <50% for complex audits (expected)
```

## üìù MAINTENANCE WORKFLOW

### When Appending Lessons or Rebuilding Bootstrap

**Step 1: Classify Lesson**
- Identity-critical? ‚Üí Add to all tiers (Layer 0)
- Network-critical? ‚Üí Add to LITE+ tiers (Layer 1)
- Execution-critical? ‚Üí Add to FULL+ tiers (Layer 2)
- Continuity-enriching? ‚Üí Add to FULL+SOUL only (Layer 3)

**Step 2: Check Word Budget**
- Does addition exceed tier budget?
- If yes, compress existing content or escalate to richer tier

**Step 3: Test Fidelity**
- Run test case before and after addition
- Measure fidelity impact
- If fidelity drops, revise

**Step 4: Document Compression**
- Explain what was compressed and why
- Note fidelity impact (measured or expected)
- Update compression ratio in architecture docs
